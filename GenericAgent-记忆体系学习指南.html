GenericAgent 记忆体系

把「越用越聪明」拆成可实现的工程：桌面上的分层档案 + 对话里的便签与锚点 + 工具闭环写入。上半篇对照真实运行机制讲透原理，下半篇用自然语言步骤、页内示意、全流程大图帮你照做（正文不嵌入程序片段）。

想象智能体的脑子是一块白板（上下文），磁盘则是带标签的抽屉柜：第一层是柜门上的便签（只写「东西在哪」），第二层是环境说明书，第三层是某类家务的「踩坑笔记」。白板写不下整本说明书，但永远知道该拉开哪一格。

下图按 GenericAgent 真实运行机制绘制。读图顺序建议：先看「一条任务的一生」彩条，再对照五条泳道。实线 = 主数据流；粉/蓝虚线 = 磁盘内容读回再注入模型；深蓝虚线 = 跨轮闭环（动画闪烁）。展开折叠块可看到每一问对应的细节说明与在系统里的大致落点（一律用自然语言表述，不贴程序标识）。

① 唤醒→ ② 多轮干活→ ③ 便签锚定→ ④ 可选结算→ ⑤ 下次见面仍认得你
① 记忆从哪来？（五种「原料」）
  • 用户话：目标与约束，本身不自动进磁盘；常记入工作要点便签，或促使模型去调用工具。
  • 工具成功结果 ★：本地执行脚本、读取文件、网页操作等已真实跑通的返回内容——这是唯一推荐写入第二层、第三层磁盘记忆的事实源（原则：没有动手验证，就不当事实写进档案）。
  • 模型自己写的单行摘要：每轮一句快照，进入回合摘要列表，再被锚点贴回对话，像一串便利贴。
  • 检查点更新：模型主动「更新工作检查点」，把要点放进内存里的工作区。
  • 外部塞条子：约定好的外部文件里写入要点或干预语，由系统读进下一轮提示，相当于人类在旁敲黑板。
② 何时处理？（像闹钟一样的钩子）
  • 进程启动：组装系统提示，把第一层索引这张「柜门便签」挂到模型眼前。
  • 每一轮工具之后：拼接工作记忆锚点，把最近摘要和工作要点便签贴回用户侧内容。
  • 第 10、20… 轮：在每轮结束回调里整段重贴第一层索引；主循环里还会周期性清空工具说明缓存，避免模型忘了工具长什么样。
  • 第 7 / 65 轮：插入强硬提醒，逼模型换策略或改为向用户提问、暂停任务。
  • 刚读完流程说明：系统会提醒：若按说明执行，记得更新工作检查点。
  • 长任务收尾：通过「开始长期记忆更新」打开专门的结算回合。
  • 后台定时：大约每十二小时，调度器把原始往来日志压缩进第四层归档，与前台对话并行。
③ 怎么处理？（下锅前的备菜）
  • 先读第零层《记忆管理规范》文档，像「厨房安全守则」。
  • 分类：环境事实→第二层；全局铁律→第一层红线区；任务专有坑→第三层。
  • 读现状再改：「读取文件」的动作 → 「按唯一旧片段替换」的改文件动作（旧文本片段 必须唯一）。
  • 并行另一条线：压缩历史消息里的长标签 / 按长度裁剪旧消息，收拾请求里附带的多轮对话，不是写磁盘。
④ 存哪里？（易失 vs 持久）
  • 关机就没：内存里的工作区、回合摘要列表、会话级消息历史——像当天工作台。
  • 落盘长期：第一层索引、第二层事实、第三层流程说明与脚本、第四层归档；另有临时目录中的模型往来日志作归档原料。
⑤ 怎么用？（模型怎么「看见」记忆）
  • 始终看见：系统说明里的第一层索引全文（地图）。
  • 每轮强化：用户侧锚点（最近摘要 + 工作要点便签）。
  • 按需深挖：用读文件工具打开第二层、第三层细节，避免把全书塞进上下文。
时机（人话） 做什么 在程序里大致对应（自然语言）

每轮工具循环 执行模型选中的动作；下一轮用户侧带上工作锚点 多轮调度主循环、具体动作分发器、以及「拼工作记忆锚点」

1. 为何需要「分层记忆」

大模型的上下文像一张越来越挤的办公桌：什么都往上堆，很快看不清重点，还贵。GenericAgent 的做法是：办公桌只留便签 + 当前文件夹，大部头书放进分层书架（磁盘）。书架上贴极短索引（第一层），正文按环境 / 任务分格（第二、三层），需要哪一页再伸手去拿（读文件工具），而不是把整库复印到桌上。

这样同一套智能体循环能跑很久，「技能」长出来的是第三层的流程说明与小脚本、第二层里经工具验证的事实，而不是把每个业务写成「如果用户说了某句就分支」的硬编码。

核心直觉：对话 = 工作记忆 + 当场推理；磁盘 = 可版本管理的长期手册。二者靠第一层指针 + 工具读写拧在一起；没有工具跑通，就不该往手册里写「事实」。
生动对比：若把全部记忆塞进系统提示，像每天背整本电话簿打电话；分层记忆则是只背人名索引，拨号前再翻页——省长度、少幻觉、多省钱。

2. 第零至四层：磁盘上的记忆拓扑

规范集中在记忆目录下的《记忆管理规范》（下文有时简称「写入规范」）。层级含义如下。

层级 典型路径 职责（一句话）
第零层 《记忆管理规范》文档本身 定义「能写什么 / 写哪 / 禁止什么」的元规范；「开始长期记忆更新」工具 会把它读给模型作结算依据。
第一层 记忆目录里的全局索引便签 极简索引（硬约束约 ≤30 行）：场景关键词指向第二、三层位置；红线与高频错规则区只压缩要点。不写长篇操作步骤、不写密钥、不写任务技术细节。
第二层 记忆目录里的全局事实库 环境特异性事实：路径、配置、凭证位置等，按分节标题组织；可随环境变长。
第三层 记忆目录下各类说明文档、可复用的小脚本 任务级可复用资产：某类任务的标准操作流程短文（关键前置 + 典型坑）、可复用脚本。只存「难重建」的要点。
第四层 归档根目录（按会话分桶） 历史会话压缩归档；原料来自 临时目录中的模型往来日志 等，由调度器周期性批处理。

第一层与第二、三层的同步规则（增删改场景、何时动红线区）也在《记忆管理规范》的表格与红线说明里；原则是：第一层只导航，不搬运细节。

2.1 补充：真实仓库里的记忆目录长什么样

下面描述的是本仓库实际采用的布局与「打开文件时你会看到什么」，便于把抽象分层对应到磁盘上的真实面貌（与个人本机长期使用后的差异见下）。

放在哪：与主程序同级、项目根下的 memory 文件夹。启动时会确保该文件夹存在；若还没有第一层索引或第二层事实文件，会从资源目录里的模板拷贝第一层草稿，或给第二层写一个只有标题行的空壳，避免首次运行报缺文件。

和 Git 的关系（很重要）：默认配置里，memory 下大部分文件会被版本控制忽略，仓库里主要显式放行的是《记忆管理规范》、各类流程说明稿、部分可复用脚本、第四层目录下的压缩脚本等。因此你克隆下来的树往往以第三层「技能稿 + 工具脚本」为主；第一层索引、第二层事实、第四层归档出来的会话包等多半只在本机跑过 Agent 之后才稳定出现，且通常不会提交到远端。

目录在磁盘上大致分这几类（自然语言树）：

  • 第零层：《记忆管理规范》稿，一篇较长的 Markdown，分段写公理、层级职责、允许与禁止事项。
  • 第一层：约定文件名为全局索引便签（仓库里常用 global_mem_insight.txt）。体量被规范压得很扁；打开后像一张「能力菜单 + 红线便签」——前几行往往是「某类场景 → 去哪个第三层说明或脚本」的极短对照；中间可能有一句提醒：需要细节时去读第二层或浏览第三层目录；再往下会列出第零、二、三、四层的指针说明（告诉模型各层文件叫什么、去哪找）；最后是一块红线规则，用编号短句写搜索习惯、编码安全、进程操作禁忌等，不写长篇操作步骤。
  • 第二层：约定文件名为全局事实库（常用 global_mem.txt）。Markdown 分节风格，按「环境路径、偏好、机器相关」等小节堆经工具验证过的事实；新机或新克隆可能几乎只有一节标题，内容随任务慢慢变厚。
  • 第三层：与记忆根目录平铺或再分子文件夹——大量 流程说明稿（任务怎么拆、常见坑）、以及 可 import 的小工具脚本（自动化某一步）；复杂能力会占一整子目录（里面还能再带说明稿与辅助脚本）。这是仓库里最容易在 Git 里直接看到的一层。
  • 第四层：通常在 L4_raw_sessions 一类目录里做会话归档；仓库里往往只跟踪「如何把原始日志压成归档」的脚本，真正归档生成的一大堆数据文件多在本机、且常被忽略规则排除。

系统提示里除了第一层全文，还会多贴一段「固定结构说明」：它来自资源目录下的短文（与第一层文件不是同一个），用几行字说明——事实库路径、流程稿与脚本在哪、第零层规范文件名、当前工作目录约定，以及几条总宪法式的行为红线（例如改记忆前要先读规范、记忆目录里优先用小补丁而不是整文件覆盖）。模型每次开任务都会同时看到这段固定说明 + 第一层便签全文，所以第一层里不必重复写目录教程，只保留索引与红线即可。

对照小技巧：若你想看「官方示例第一层长什么样」，可读资源目录里的第一层模板（与运行时第一层同名但带 template 字样）；运行时第一层是在此基础上、由你与 Agent 长期协作改出来的「个人版菜单」。具体落盘内容请以本机记忆目录与工具执行结果为准，本文不附可点开的示例文件正文。

3. 四大公理与工程含义

《记忆管理规范》开头的公理，直接决定「智能体能不能把胡编写进磁盘」。

3.1 行动验证（只认跑通的结果）

写入一至三层的内容须能追溯到成功的工具结果（如读文件确认、本地命令执行成功）。禁止把模型臆测、未执行计划当事实。一句话：没有动手验证，就不当事实写进档案。

工程含义：「自我进化」不是后台静默从日志抽技能，而是模型在结算轮里按写入规范用「按唯一旧片段替换」等改文件动作主动、可审计地改磁盘。

3.2 神圣不可删改（对已验证数据）

重构或清理时不可丢准确性；可压缩、可迁层（如事实迁到任务说明），但信息要可追溯。操作上偏好小步补丁，避免整文件覆盖。

3.3 禁止易变状态

不存进程号、临时会话标识、易变绝对路径等。否则记忆很快「腐烂」，还会误导后续任务。

3.4 最小充分指针

上层索引只保留能定位到下层的最短关键词；多写一词就多占第一层行数预算、多引入噪声。

4. 「这条信息放哪？」决策树

记忆流程规范用决策树归纳；掌握它可避免「全塞进事实库」或「把常识当宝贝存」。

  • 环境特异、模型零样本猜不准的事实 → 第二层，高频场景再在第一层加「关键词 → 目标位置」指针。
  • 全局操作规律、与具体任务无关 → 压成一句进第一层红线区。
  • 某任务特有、踩坑代价高、未来还要用 → 第三层（流程说明或可执行小脚本）。
  • 通用常识或几次探测就能重建的信息 → 不存。

5. 启动时：系统提示如何注入第一层索引

主入口里组装系统提示 读取 角色与行为总则模板，追加日期，再追加 读取全局记忆索引供提示使用。

读取全局索引块 会读取：

  • 记忆目录里的全局索引便签（第一层全文），
  • 说明记忆目录结构的模板文（目录结构说明），
  • 并提示工作目录 临时工作目录 路径。

因此每次开任务，模型都「看见」当前第一层索引与记忆树形状；细节仍在磁盘，按需读文件。

首次运行：应用主入口若发现没有全局事实库文件或全局索引便签文件，会创建空文件或从模板拷贝索引草稿。

6. 任务内：处理器与工作记忆

通用智能体动作处理器（动作与记忆工具实现文件）维护两类与「当前任务」强相关的状态：

  • 当前任务工作区字典：如工作要点便签（短记事本）、建议重读的操作说明名、是否处于分步计划模式（及计划文件路径）等。
  • 摘要链列表：字符串列表；每轮结束从回复里的「单行摘要」约定段或工具摘要里再记一行，形成人类可读的回合摘要链。

6.1 锚点注入

多数工具在返回前会拼接工作记忆锚点；同一轮里若连续多个工具，通常只对第一个附完整锚点，避免重复。注入内容包括：

  • 最近约 20 条回合摘要，包在「历史摘录」区块里；
  • 当前是第几轮；
  • 若有：工作要点便签、建议重读哪份操作说明。

这相当于在不无限拉长单次请求里的多轮历史的前提下，把「任务内工作记忆」反复贴回模型输入侧。

6.2 更新工作检查点

工具 「更新工作检查点」工具 更新 工作要点便签 / 建议重读的操作说明名，并把 跨对话继承计数 置 0。用于长任务中段保存约束与进度，避免上下文漂移。

7. 轮次节奏：每轮结束时的回调

每轮工具执行完，「每轮结束回调」会做这些事：

  • 解析模型是否写了单行摘要，写入回合摘要列表；没有就提醒补上。
  • 第 65 轮（且不在分步计划特殊模式）：强烈建议改为向用户提问、暂停瞎转。
  • 第 7 轮倍数：提醒禁止无新信息的死磕，可更新工作检查点。
  • 第 10 轮倍数：再次贴上整段第一层索引，防止长任务忘地图。
  • 分步计划模式：周期性要求重新阅读计划文件核对进度；轮数过高则必须问用户。
  • 若任务目录里有外部塞条子的文件，把内容合并进工作区或下一轮提示，并标成来自操作者。
记忆相关知识点：「每 10 轮刷第一层索引」不是换模型，而是防止长任务忘记全局地图；它与「每十轮强制重贴完整工具说明书」是两套不同的防膨胀手段，互不替代。

8. 长期结算：「开始长期记忆更新」

当模型调用「开始长期记忆更新」，就进入记忆结算回合：

返回《记忆管理规范》全文（若存在），作为第零层规范。
拼接固定中文指令：提炼经行动验证的事实与经验，按类型写入第二、三层，并维护第一层；强调先读文件再尽量用「按唯一旧片段替换」做小改动。
再次附上 读取全局索引块，与当前全局索引对齐。

工具描述（含中文版参数约定）还要求超长任务（如 15+ 轮）结束后必须调用，把「记得沉淀」产品化为模型义务。注意：没有独立后台进程自动写技能说明；沉淀质量依赖模型是否遵守流程规范与公理。

9. 主循环与会话历史

多轮工具主循环每一轮都会带工具列表去对话。第一轮消息里含系统角色说明和用户原话；之后每一轮往往只带本轮用户补充和各工具刚返回的结果——更长的往来记在会话对象内部，不全部重复塞进单轮请求。

另有策略：每十轮清空「工具说明缓存」，迫使下一轮重新附带完整工具说明书（与「每十轮重贴第一层索引」互补：一个防忘工具长什么样，一个防忘地图）。若单轮拼出来的提示过长，也会清空该缓存，避免模型以为还能调用工具却看不到说明书。

在原生工具调用路径下，底层会话接口把用户块追加进会话内部保存的对话链，再追加助手返回的结构化内容，与厂商接口对齐。因此「对话记忆」的权威副本在大模型连接模块里的会话对象，而不是主循环函数自己的临时列表。

文本协议工具路径则把多轮内容拼成一大段约定格式的文字；多轮怎么堆叠取决于底层提问接口，与原生块结构不同。实际产品里常用原生路径。

9.1 多后端切换时的 history

轮换大模型客户端时，会把上一客户端里保存的完整对话链交给新客户端，保证对话线程不丢。若接入层拼装不当，这里会报错提示检查密钥与模型配置。这与磁盘一至四层无关，但属于「会话级记忆」连续性的关键实现。

10. 对话历史压缩与裁剪（省上下文长度）

压缩历史消息里的长标签：按间隔对较早消息里的思考段、工具调用段、工具结果段以及历史摘录、工作要点等大段做截断，只留头尾、中间用省略表示。

按长度裁剪旧消息：当序列化长度超过阈值时，更激进压缩并从头部删消息，且保证从用户轮开始裁剪，必要时清理首条用户消息里悬空的工具调用痕迹。

这与磁盘第一层「控制索引行数」是同一哲学在内存与单次请求上下文上的映射。

11. 第四层归档与定时调度

写入模型往来日志等步骤，会把一问一答追加到临时目录下的模型响应日志里。

定时调度模块的调度入口里，约每 12 小时调用「会话归档批处理」，把原始往来日志批处理进第四层。这是与主智能体循环解耦的归档线，用于事后检索过往上下文，而不是替代一至三层的精炼知识。

12. 工具与记忆的契约

工具与记忆的关系
「读取文件」的动作读第二、三层或流程说明；路径落在记忆或流程目录时会提示结合规范更新工作记忆；并可记录记忆文件访问统计。
「按唯一旧片段替换」的改文件动作 / 「写入或追加文件」的动作实际修改磁盘记忆与代码；长期结算主要依赖二者（加 从正文引用展开成片段）。
「更新工作检查点」工具只改处理器内内存里的工作区，不直接写第二层、第三层磁盘文件。
「开始长期记忆更新」工具触发「读写入规范 + 固定指令」的结算回合，驱动后续写一至三层磁盘。
「在本地执行脚本或命令」的动作不自动写记忆；可运行自定义脚本处理环境，但记忆流程规范倾向小补丁改文件，而不是大范围改库式编辑。

13. 跨会话与飞书场景

新任务创建新处理器实例时，应用主入口可把上一任务的工作要点便签拷入，并增加跨对话继承计数与系统提醒：若已是新任务应更新或清除工作记忆，避免旧任务记事污染新任务。

飞书来源且已有历史时，首轮用户内容会带上 拼接工作记忆锚点 再拼当前用户句，保证多消息会话仍带锚点。

14. 常见误区与边界

  • 「自动进化」≠ 无模型参与：磁盘写入是模型在工具与写入规范约束下完成的，不是单独的离线挖掘管道。
  • 「技能树」主要是第一层索引 + 多份第三层文件，不是图数据库；检索靠关键词与读文件工具。
  • 可选的「技能卡检索」扩展是对外接口检索现成技能卡，与「本次任务自动沉淀到第三层」是两条线。
  • 安全：本地执行命令与写文件能力极强，记忆文件勿存密钥；公理与第一层红线也明确禁止。

下篇 — 小白动手实现

16. 你要实现的「最小记忆系统」长什么样

若目标是「像 GenericAgent 一样越用越强」，最小可交付不必一次做全第四层归档、对话历史压缩、多前端。你只要打通下面闭环即可：

  1. 磁盘：至少三个文件——第一层索引、第二层事实、第零层写入规范；再加一个第三层目录（放每个任务的短流程说明）。
  2. 启动：把第一层索引（加目录结构说明）拼进系统提示，让模型知道「记忆放哪、叫什么」。
  3. 会话内：内存里维护 工作要点便签 + 最近 N 条单行摘要；每轮工具后把这两块追加进用户侧提示（锚点）。
  4. 工具：给模型三个动作——更新工作检查点、读文件、唯一块补丁写入（或只允许往第二层某一节末尾追加）。
  5. 结算：任务结束前插入一轮「读第零层规范 + 再读第一层索引 + 只允许小块补丁写第二、三层并维护第一层」——与 GA 的「开始长期记忆更新」工具同构。
小白易错点：不要一上来就做向量库 / 图数据库。GA 的「技能树」本质是文件系统 + 短索引 + 模型按需读文件。先把文件与补丁式修改写对，再考虑检索增强。

17. 第一步：目录与四个文件的初始内容

在你的项目根下建记忆根文件夹（名字可自定，但全程要固定）：

  • 第一层索引文件：极短，硬性控制行数（例如不超过约四十行）。
  • 第二层事实文件：用「二级标题分节」组织环境事实。
  • 第零层规范文件：给人和模型共同遵守的写入规范（Markdown）。
  • 第三层技能目录：下面再放具体任务的短流程说明，例如「某业务的极简踩坑笔记」。

第一层要写什么：场景关键词指向某份第三层说明或第二层某一节；高频场景写清「去哪看」；低频只列关键词；另附几条一行一条的全局红线（例如禁止把密钥写进记忆文件）。

第二层要写什么：开头标明这是全局事实库；分「路径与环境」「用户偏好」等小节；只收录经工具验证过的内容。

第零层最小规范：三条就能启动——① 只写已验证事实；② 第一层只加索引不写长篇操作步骤；③ 修改磁盘文件时用「唯一旧片段替换」，禁止整文件瞎覆盖。

18. 第二步：读第一层索引并进系统提示

对应 GA 的「读取全局索引块」。你自己实现时做两件事即可：

  1. 读磁盘上第一层索引文件的全文；再拼上几行固定不变的目录约定说明（告诉模型各层文件叫什么、各管什么）。
  2. 把「角色与行为说明」放在最前，后面接上上一步拼好的地图块，整体作为每次调用大模型时的系统侧说明。

不要把第二层事实全文默认塞进系统说明——会爆长度；需要时再让模型用读文件工具按需打开。

19. 第三步：会话内状态与锚点字符串

对应 GA 处理器里的工作区与回合摘要列表。建议在内存里用一个小结构存四样：

  • 工作要点便签（短字符串）——当前约束、进度、易忘参数。
  • 回合摘要列表（字符串列表）——每轮模型输出的一行摘要，可用协议强制模型必须写。
  • 关联流程说明（可选路径）——提醒模型必要时重读第三层某份说明。
  • 当前轮次（整数）——便于做「每十轮重贴第一层索引」等策略。

锚点怎么拼（与 GA 的「工作记忆锚点」同构）：取最近约二十条摘要连成一段，包在「历史摘录」区块里；写上当前第几轮；若有工作要点便签，再包一层「工作要点」区块；若有流程说明路径，附一句「必要时请重读某文件」。

把这段锚点拼在用户消息前（或拼在工具结果之后、下一轮用户内容里），长任务才不容易丢约束。

20. 第四步：一轮里用户消息怎么拼（时序）

简化后的时序（多工具可并行扩展）：

  1. 首轮：系统侧用上一节拼好的完整系统说明；用户侧用「锚点 + 空行 + 用户原话」。
  2. 模型返回 tool calls → 你在本地执行工具 → 得到字符串结果。
  3. 下一轮用户侧内容：通常包含「工具返回的结构化或纯文本结果」+ 组装锚点段落（锚点要反映最新工作要点便签与摘要）。
  4. 若模型调用了 更新检查点，先更新内存里的 工作要点便签，再进下一轮。
  5. 任务将结束时：先发一条「结算指令」用户消息（见下一节），让模型只做读文件和小块补丁写入。

GenericAgent 里多轮往来存在会话级消息历史；你自己对接各厂商提供的客户端库时，把上述内容拆成多条角色消息即可，原理相同。

21. 第五步：三个必备工具（参数与处理逻辑）

与 GA 的九工具相比，最小可交付版本只需这三个与记忆强相关（其余业务工具另加）：

工具名参数（概念）你的处理器做什么
「更新工作检查点」工具工作要点便签；可选建议重读的操作说明名只写会话内结构体，不碰磁盘。
受控的读记忆文件文件路径；可选起止行按路径读出文本；建议限制只能读记忆根目录下的文件。
受控的补丁式改文件路径；要被替换的旧片段；替换后的新内容做唯一块替换（见 §23）；失败则把原因返回给模型。

若用大模型厂商的「函数调用」能力：为每个工具写清名称、参数类型与是否必填即可；更新检查点至少必填「工作要点」字符串，「关联流程说明」可选。

补丁式改工具的说明里务必写清：旧片段必须在目标文件中出现且只出现一次，否则让模型先读文件再改。

22. 第六步：长期结算回合

GA 的做法是：工具处理器把《记忆管理规范》全文和一段固定「结算指令」一并交给模型，作为下一轮用户侧内容的一部分。你照这个结构即可：

  1. 读出第零层规范文件，作为工具返回或用户文本交给模型。
  2. 固定指令里写死：先读目标记忆文件、再用补丁式修改、禁止写密钥、第一层只加索引行。
  3. 再次附上当前第一层索引全文（与启动时同构的那块地图），避免模型覆盖错。
  4. 本轮之后尽量不要再开复杂业务工具，减少边结算边乱改的概率。

触发时机自定：例如用户点「结束任务」、轮次超过某阈值、或模型主动调用你提供的「提交长期记忆」类工具（可与 GA 的「开始长期记忆更新」同职责）。

23. 第七步：补丁式改文件——唯一块替换（核心）

这是 GA「按唯一旧片段替换」的灵魂，用来避免模型整文件覆盖把资料库冲烂。逻辑用自然语言说就是：

  1. 读出目标文件全文；
  2. 数一下「旧片段」出现了几次：零次则报错「找不到」；多于一次则报错「请加长旧片段以保证唯一」；
  3. 恰好一次时，只做一次替换，写回磁盘，返回成功。

进阶可做：正文里写「某文件某行范围」自动展开成真实片段（GA 有类似能力）、路径白名单、改前 diff 给用户确认。

24. 第八步：主循环里五件要记住的事

把下面五件事挂在你的智能体主循环里，就与 GA 记忆逻辑同构：

  1. 每轮开始：轮次加一；可选每满十轮把第一层索引再拼进用户侧（或刷新工具说明书缓存）。
  2. 模型输出后：解析单行摘要约定，追加进回合摘要列表。
  3. 工具返回前：同一轮多个工具并行时，通常只对第一个附完整锚点，避免重复（GA 用「第几个工具」去重）。
  4. 危险轮次：例如每 7 轮注入「禁止无效重试」类提示（可借鉴 GA 文案思路）。
  5. 任务结束：走 §22 结算回合。

25. 仓库里的入门演示脚本

文档同目录的 examples 文件夹里有一份不依赖 GA 其余模块的 Python 入门脚本，用来把上文步骤串成可跑通的一遍。它大致包含这些能力（名字随仓库为准，此处只说明职责）：

  • 初始化第零到三层目录与占位文件；
  • 读取第一层索引并拼系统提示；
  • 会话内维护检查点与锚点字符串；
  • 教学用的「一条事实该进哪一层」分类示意；
  • 对第二层文件做一次唯一块替换的演示；
  • 拼出结算回合里交给模型的两段文字（规范 + 固定指令）。

在本机用 Python 运行该脚本后，终端会依次打印：带第一层全文的系统提示片段、一轮典型的锚点长什么样、分层路由的判定示例、补丁写入第二层后的片段、以及结算指令前缀有多长。演示往往在临时目录里落盘，跑完即删；你把思路迁到自己项目时，把记忆根目录改成固定路径即可长期保留。

25+. 演示跑一遍时会发生什么（文字版）

不看终端也能理解脚本在做什么，按顺序对应上面五段输出：

  1. 系统提示：角色说明后接上「目录约定」，再贴上第一层索引全文，让模型一开始就拿到地图。
  2. 锚点：出现「工作记忆」标题；历史摘录里是多轮单行摘要；标明当前轮次；工作要点里写用户约束；必要时提醒重读某份第三层说明。
  3. 分层路由：用标签区分环境事实、全局红线、应丢弃的易变状态、任务专有流程，便于对照 §4 决策树。
  4. 写第二层：演示唯一块替换成功后，第二层里出现经「列表目录」等验证过的路径一类事实。
  5. 结算：先量一下第零层规范有多长，再附上固定话术——只写已验证内容、禁密钥、先读再改、第一层只加索引——并再次贴上第一层全文防误删。

若你本地运行结果与教学文字有出入，以仓库里脚本当前行为为准；本文只描述结构与学习顺序。

  • 第一层索引是否控制在约定行数内？是否仍出现大段操作步骤？
  • 系统说明是否只带第一层索引与目录结构，而不是第二层全文？
  • 锚点是否每轮（或每轮首个工具后）更新？
  • 受控的补丁式改文件 是否强制唯一 old 块？
  • 是否禁止模型把访问密钥写入记忆目录？（工具说明里写明 + 服务端校验路径与内容）
  • 结算回合是否与业务工具隔离？
  • 新任务开始时是否清空或继承 工作要点便签（策略与 GA 一致即可）？

从最小可交付到完整 GenericAgent 还差什么（按需再做）：多模型原生工具协议、对话历史压缩与裁剪、第四层定时归档、飞书等多前端、分步计划模式、记忆文件访问统计、以及完整的九原子工具与浏览器驱动。这些都不是「记忆闭环」的必要条件。
